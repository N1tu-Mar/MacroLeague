  How MacroLeague is built — a ground-up tour

  1. The big picture: two halves that talk over the internet

  Almost every modern app is split into two halves:

  - Front end (the "client") — the app that runs on the user's phone. In your
  case this is the code in src/, written in React Native + Expo. It draws
  screens, handles taps, and shows numbers. It is the waiter in a restaurant: it
  takes orders and brings food, but it doesn't own the kitchen.
  - Back end (the "server") — code and data that run on computers you control in
  the cloud. In your case this is Supabase. It's the kitchen and the safe: it
  stores all the real data, enforces the rules, and hands out food only to
  people allowed to have it.

  They talk to each other over the internet by sending small messages (requests
  and responses), usually carrying JSON (a text format for structured data).

  Why split it this way? Two reasons that matter for you specifically:

  1. You can never trust the phone. A determined user can open up the app on
  their phone and lie to it — claim they have 1,000,000 XP, claim a meal had 0
  calories, etc. So the real rules and real numbers must live on the server
  where the user can't touch them. This single idea drives almost every design
  decision in your codebase.
  2. Everyone shares one source of truth. The leaderboard only works if all
  phones read from the same database. If each phone kept its own scores, there'd
  be no competition.

  ---
  2. The front end, layer by layer

  Your src/ folder is organized in layers, from "what the user sees" down to
  "how we talk to the server." This separation is deliberate and good.

  - src/screens/ — full screens (Home, MealLogger, Leaderboard, Profile…). These
  are what the user looks at.
  - src/components/ — reusable building blocks (a MacroRing, a StreakFlame, a
  LeaderboardRow). Screens are assembled out of these, like Lego bricks.
  - src/navigation/ — the rules for moving between screens. App.tsx decides: if
  you're logged in, show MainNavigator (the tabs); if not, show AuthNavigator
  (sign-in/sign-up).
  - src/store/userStore.ts — short-term memory for the running app (using a
  library called Zustand). It holds "who is logged in" and their current
  XP/level so every screen can read it without re-fetching. Think of it as a
  whiteboard the whole app can glance at.
  - src/services/ — the only layer allowed to talk to the back end.
  mealLogService, leaderboardService, challengeService, etc. Each function here
  is a specific conversation with Supabase ("save this meal", "get the
  leaderboard").
  - src/lib/supabase.ts — the actual phone line to Supabase. It's configured
  once and shared everywhere.

  Why layer it like this? When a screen needs data, it calls a service, and the
  service calls Supabase. The screen never builds a database query itself. This
  means if you ever change how meals are stored, you fix it in one service file,
  and every screen keeps working. This is the single most important habit for
  keeping an app maintainable as it grows.

  ---
  3. What Supabase actually is, and the databases inside it

  Supabase is a hosted bundle of back-end tools. You're using three of its
  pieces:

  a) Auth — identity

  Handles sign-up, login, Google sign-in, and passwords. When someone logs in,
  Supabase hands the phone a JWT — a signed, tamper-proof ID badge (a "JSON Web
  Token"). The phone shows this badge on every request so the server knows who
  is asking. Critically, the user can't forge it — it's cryptographically signed
  by Supabase.

  b) Postgres — the database (the "tables")

  This is the heart of everything. Postgres is a relational database: data lives
  in tables (like spreadsheets) that reference each other. Here are your tables
  and why each one exists:

  Table: profiles
  What it holds: One row per user: username, macro goals, XP, points, streak, 
    level counters
  Why it matters: The user's identity + their game stats. Linked 1-to-1 to the
    Auth user.
  ────────────────────────────────────────
  Table: meal_logs
  What it holds: One row per logged meal (calories, protein, etc.)
  Why it matters: The core event of the whole app. Everything else is downstream

    of this.
  ────────────────────────────────────────
  Table: foods
  What it holds: A cache of food definitions (manual + USDA)
  Why it matters: So you don't re-fetch the same food's nutrition forever.
  ────────────────────────────────────────
  Table: nutrition_sources
  What it holds: A tiny list: usda_fdc, manual, user_estimate
  Why it matters: Records where a food's numbers came from (provenance).
  ────────────────────────────────────────
  Table: food_search_cache
  What it holds: Raw USDA search responses, keyed by query
  Why it matters: Saves money + avoids USDA rate limits (more below).
  ────────────────────────────────────────
  Table: food_portions
  What it holds: Household serving sizes ("1 slice = 107g")
  Why it matters: For future portion scaling.
  ────────────────────────────────────────
  Table: gamification_events
  What it holds: An append-only ledger — one row per XP/points movement
  Why it matters: The true source of truth for scores. (Explained below — this
  is
     the cleverest part.)
  ────────────────────────────────────────
  Table: user_daily_activity
  What it holds: One row per user per day (meals, macros, streak flags)
  Why it matters: Powers streaks and "did you hit your goal today" rules.
  ────────────────────────────────────────
  Table: gamification_rule_sets
  What it holds: The scoring rules stored as data (JSON)
  Why it matters: Lets you tune the game without changing code.
  ────────────────────────────────────────
  Table: rewards / user_rewards
  What it holds: Reward catalog + who redeemed what
  Why it matters: The "spend your points" half of the loop.
  ────────────────────────────────────────
  Table: challenges / challenge_participants / challenge_goals
  What it holds: Competitions between users
  Why it matters: The social/competitive layer.

  A note on the word "databases" (plural): you actually have one database
  (Postgres) containing many tables. People loosely say "databases" when they
  mean tables. The tables are connected to each other by foreign keys — e.g.
  meal_logs.user_id points at profiles.id. That's what makes it "relational":
  rows in one table are linked to rows in another.

  c) The critical safety net: Row Level Security (RLS)

  This is the concept that makes the whole thing safe, so it's worth
  understanding deeply.

  Normally a database is all-or-nothing: if you can read a table, you can read
  every row. That would be a disaster — anyone could read everyone's meals. RLS
  changes that. It's a rule attached to each table that says, per row, "can this
  user touch this row?"

  Look at this from your 0001 migration:

  create policy "read own meals"
    on public.meal_logs
    for select
    using (user_id = auth.uid());

  In plain English: "You may only SELECT meal rows where the row's user_id 
  equals your own logged-in user id." auth.uid() is the server reading the ID
  off that tamper-proof JWT badge. So even though every user's meals sit in the
  same table, the database itself refuses to show you anyone else's. The 
  security is enforced by the database, not by the app — which is exactly right,
  because you can't trust the app.

  4. The USDA data — and an important correction

  You asked "how we scraped all the USDA data." Here's the thing: the app does 
  not scrape or bulk-download USDA data at all — and that's a deliberate, good
  decision. Let me explain what actually happens, because the difference
  matters.

  USDA runs FoodData Central (FDC), a free U.S. government database of food
  nutrition. There are two ways to use it:

  - Bulk scrape: download all ~600,000 foods into your own database. Heavy, goes
  stale, lots of storage, and most of it you'll never use.
  - On-demand + cache (what you do): when a user types "grilled chicken breast,"
  you ask USDA just for that, then save the answer so the next person who types
  it gets an instant hit without calling USDA again.

  Here's the flow (from supabase/functions/estimate-meal/):

  User types "grilled chicken breast" in the app
     │
     ▼
  src/services/nutrition/mealEstimateService.ts   (front end)
     │   sends the text + the user's JWT badge
     ▼
  estimate-meal  EDGE FUNCTION  (server code, runs in Supabase)
     │   1. Is this query already in food_search_cache, not expired? → return it
  (no USDA call)
     │   2. Cache miss → call USDA FoodData Central with the SECRET api key
     │   3. Convert USDA's per-100g nutrients → per-serving macros
     │   4. Save the foods + cache the search result (7-day expiry)
     ▼
  Returns a list of candidate foods, each EDITABLE
     │
     ▼
  User picks one, edits the numbers if needed, taps SAVE → logMeal() → meal_logs

  What's an "Edge Function"? It's a small piece of your code that runs on
  Supabase's servers, not on the phone. It exists for one crucial reason: the 
  USDA API key is a secret. If you put that key in the phone app, anyone could
  extract it and abuse it. So the key lives only on the server
  (Deno.env.get('USDA_FDC_API_KEY')), and the phone is physically incapable of
  seeing it. The phone asks the edge function; the edge function holds the key
  and talks to USDA. This is the same pattern you'll use for any secret (OpenAI,
  payment keys, etc.).

  Why cache? USDA limits how many times per hour you can call it (rate limits),
  and network calls are slow. The cache (food_search_cache, 7-day expiry) means
  the 100th person to log "chicken breast" gets an instant answer and you never
  hit USDA. The docs in docs/nutrition-architecture.md explain this reasoning
  well.

  One honest limitation baked into the design: USDA matches single foods, not
  whole sentences. "Pizza with pineapple and ham" returns pizza-ish candidates;
  it doesn't split the sentence into pizza + pineapple + ham and add them up —
  unless you turn on the optional OpenAI "composite" parser (which only extracts
  the ingredient list; all the actual nutrition numbers still come from USDA,
  never the AI). That's why the whole flow is framed as "estimate, then let the 
  user correct it" rather than "perfect parser." That's a smart, honest choice —
  nutrition apps that pretend to be perfect frustrate users.

  5. How a meal gets logged — and the genius of how scoring works

  This is where the architecture is genuinely impressive, so I'll slow down.

  Step 1: The save

  When you save a meal, src/services/mealLogService.ts → logMeal() validates the
  numbers (no negative calories, valid meal type, etc.) and inserts one row
  into meal_logs. There's one save path for every kind of meal (manual, USDA,
  composite) — they all become a meal_logs row. Simplicity = fewer bugs.

  Two safety features worth understanding:

  - Idempotency (client_request_id): the phone generates a unique ID for each
  save. The database has a unique index on (user_id, client_request_id). If the
  network hiccups and the phone retries the same save, the database rejects the
  duplicate — so a flaky connection can never log your lunch twice. The code
  catches that specific error and just returns the already-saved meal. This is a
  professional-grade touch most prototypes miss.
  - Snapshotting: the macros are copied into the meal_logs row at save time. If
  a food definition later changes, your history doesn't silently rewrite itself.
  Yesterday's lunch stays what you actually ate.

  Step 2: The scoring (the part you should be proud of)

  Here's the question: when you log a meal and earn +50 XP, who decides that, 
  and where?

  The naive (wrong) answer: the phone adds 50 to your XP and tells the database
  "set my XP to the new number." This is catastrophic — a user could just say
  "set my XP to 9,999,999."

  Your app does it the right way. When a row is inserted into meal_logs, the
  database itself fires a trigger — a piece of code (award_meal_gamification())
  that runs automatically inside the database, in the same transaction. The
  trigger:

  1. Writes a row into the gamification_events ledger (+50 XP, +10 points).
  2. Updates the day's user_daily_activity (meal count, running macros).
  3. Advances the streak if it's a new day.
  4. Checks the rule set — did you hit your protein goal? 3-meal goal? — and
  awards bonuses.
  5. Rolls the totals up onto your profiles row.

  The phone never writes any of these numbers. In fact, migration 0005 does
  this:

  revoke update on public.profiles from anon, authenticated;
  grant update (username, timezone, goal_calories, ...) on public.profiles to
  authenticated;

  Translation: "Users may update their username and goals — but they are 
  physically forbidden from writing the xp, points, or streak columns." Even if
  a hacker bypassed your app entirely and talked to the database directly with a
  valid login, the database would reject any attempt to change their own score.
  That is exactly how it should be, and it's a level of rigor well above a
  typical demo.

  Why the ledger (gamification_events)? Your profiles.xp is just a cached 
  running total. The real truth is the append-only ledger — one row per points
  movement, never edited. This means:
  - You can always recompute anyone's balance by summing their ledger. If a bug
  ever corrupts the cached total, you can rebuild it.
  - The leaderboard is just "sum everyone's ledger over the last 14 days"
  (get_leaderboard).
  - Redeeming a reward writes a negative ledger row, so spending points is
  auditable too.

  This is literally how banks and accounting systems work. It's the correct
  foundation for anything involving points that might one day have real value.

  The whole loop

  log meal → trigger awards XP/points → streak advances → leaderboard_delta
  added
     → climb leaderboard / challenge standings → spend points on rewards

  ---
  6. So — why does it need to work this way?

  Pulling the threads together, every major decision traces back to one of three
  forces:

  1. You can't trust the client. → Scores are computed and stored server-side by
  a database trigger; the client is revoked from writing them; RLS stops users
  reading each other's data; secret API keys live only in edge functions.
  2. Everyone shares one truth. → A single Postgres database with one meal_logs
  table and one ledger; the leaderboard reads the same ledger everyone writes
  to.
  3. History must stay honest. → Append-only ledger, snapshotted macros,
  idempotent inserts. Nothing silently rewrites the past.

  A weekend-prototype version of this app would put scores in the phone and
  trust them. Yours doesn't, and that's the difference between a demo and
  something you could actually ship money-backed rewards on.

  What I'd do next — researched recommendations

  I read all the planning docs (Next steps, the two Claude Code prompts files,
  the .plan.md files, building.md, Databaseprompt.md). Here's my prioritized
  take, with reasoning. I've grouped them by why.

  Tier 1 — Correctness gaps that will bite real users

  1. Onboarding doesn't save the profile. Sign-up creates an Auth user, and a
  trigger makes a bare profiles row with a placeholder username (user_xxxxxx),
  but the name/university/goal type/macro targets the user enters during sign-up
  are not persisted. Look at App.tsx — it hard-codes university: 'Rutgers 
  University' and goalType: 'muscle' for everyone. Why it matters: until you
  save onboarding into profiles, macro goals are fake, the macro-accuracy rule
  can't fire correctly, and the leaderboard shows everyone at the same school.
  This is the highest-leverage fix. Where: a SignUpScreen → updateProfile call
  after the auth user is created.
  2. Edit/delete a meal does NOT reverse the XP. This is documented as
  deliberate (the trigger fires on INSERT only), and it's a fine MVP choice —
  but it means a user can log a meal, earn 50 XP, delete it, and keep the XP.
  Why it matters: the moment rewards have any real-world value, this is an
  exploit. Before any real reward launch, you need clawback logic (an
  UPDATE/DELETE trigger that writes a reversing ledger row). For now, just know
  it's there.

  Tier 2 — Closing the loop the docs promised

  3. Challenges have no winner. challenges_won is always 0 — there's no
  finalization step that declares a winner when a challenge ends. The schema is
  ready for it (event_type = 'challenge_win' exists). A scheduled job (Supabase
  cron) that finalizes ended challenges would complete the competitive loop.
  4. No "friends" / social graph. The Friends and My-Team leaderboard tabs are
  honest "coming soon" placeholders because there's no friendships table. If
  campus-vs-campus or friend competition is core to your vision, this is the
  missing table.
  5. Meal-logging input modes. Right now it's manual + describe-in-text. The
  docs call for photo scanning, food search, and one-tap dining-hall logging.
  Photo scanning (an analyze-food-photo edge function calling a vision model —
  key stays server-side) is the highest-wow feature for a campus app.

  Tier 3 — Before you call it "production"

  6. Automated tests for the money-critical paths. You have SQL tests for
  migrations (supabase/tests/), which is great. But the scoring logic — the part
  where a bug literally creates or destroys points — deserves the most testing.
  I'd write tests that: log meals across a timezone boundary, retry the same
  client_request_id, and try to redeem a reward you can't afford, asserting the
  ledger stays consistent.

  How to make sure the app is secure and "good to use"

  You wrote "make sure the app is vulnerable" — I'm reading that as the
  opposite: make sure it's not vulnerable. Here's what to actually look into.
  The good news: your foundation is already strong. These are the things to
  verify, in plain terms:

  Security — what to check (and how)

  1. Run Supabase's own linter. Supabase has a built-in Security Advisor in the
  dashboard (and supabase db lint). It flags tables with RLS off, functions with
  mutable search_path, etc. Run it — it's free and catches the common mistakes
  automatically.
  2. Confirm RLS is on for every table. Your migrations enable it consistently,
  but a single missed enable row level security on a future table = a public
  data leak. Make this a checklist item for every new table.
  3. Verify the anon key is the anon key, never the service-role key. Your .env
  correctly uses EXPO_PUBLIC_SUPABASE_ANON_KEY. The anon key is meant to be
  public (RLS protects you). The service_role key bypasses RLS entirely and must
  never appear in src/, .env as a public var, or git. (It's correctly absent.
  The .env is also correctly git-ignored and untracked — I checked.) Keep it
  that way.
  4. Rate-limiting and abuse. The estimate-meal function requires a logged-in
  user (good), but a logged-in user could spam it. Supabase has edge-function
  and auth rate limits — review them before launch. Same for meal logging if
  points ever become valuable.
  5. The reward race condition is already handled — redeem_reward locks the
  profile row (for update) so two simultaneous redemptions can't both succeed on
  insufficient points. This is a subtle bug class your code already got right;
  just don't regress it.

  "Good to use" — the quality/reliability layer

  6. Error handling that reaches the user. Your describeDbError already turns
  cryptic Postgres errors into human messages ("Your profile is not set up
  yet…"). Extend that habit to every screen so a failed save never silently does
  nothing.
  7. Loading and empty states. A fast app feels trustworthy. Make sure every
  data fetch shows a spinner and every empty list (no meals yet, no challenges)
  shows a friendly placeholder, not a blank screen.
  8. Timezones. Streaks depend on "what local day is it," and your code handles
  this carefully (getUtcDayRange, profile timezone in the trigger). Test it
  deliberately: log a meal at 11:50 PM and 12:10 AM and confirm streaks behave.
  Timezone bugs are the #1 source of "my streak broke!" complaints in habit
  apps.
  9. Accessibility & device testing. Test on a real cheap Android phone, not
  just the simulator. Font sizes, tap targets, and slow networks behave
  differently there.

  Here's what I built for soft-delete-with-recovery.

  The flow it implements

  User taps "Delete Account" (Profile)
    → account-lifecycle edge function (authed as the user)
        → request_account_deletion() RPC sets deactivated_at +
  deletion_scheduled_at (+14 days)
        → emails a heads-up ("deleting on <date> — sign in to cancel")
    → app immediately shows the Reactivate gate (no normal app access, hidden
  from leaderboards)

  Within 14 days, user signs in → sees the gate → "Reactivate my account"
    → reactivate_account() RPC clears the flags → everything is exactly as they
  left it

  After 14 days → purge-accounts (scheduled, service-role) permanently deletes
  the auth user
    → ON DELETE CASCADE removes all their rows; shared USDA food cache is kept

  This directly satisfies your two requirements: a stolen-password attacker can
  only archive the account (the real owner gets an email and recovers it), and a
  user who changes their mind just signs back in.

  Files

  Backend
  - supabase/migrations/0009_account_soft_delete.sql — two flag columns on
  profiles, the request_account_deletion() / reactivate_account() SECURITY
  DEFINER RPCs (each acts only on auth.uid(); the columns are not
  client-writable, so deletion can only go through these), and
  leaderboard/challenge functions updated to hide archived users.
  - supabase/functions/account-lifecycle/ — the in-app deactivate/reactivate
  edge function you asked for; runs as the user, sends the email.
  - supabase/functions/purge-accounts/ — the scheduled hard-delete, protected by
  a secret header (not user-callable).
  - supabase/config.toml — registered both functions (account-lifecycle
  JWT-required; purge-accounts JWT off, secret-gated).

  Client — accountService.ts, ReactivateAccountScreen.tsx, plus wiring in
  App.tsx, userStore.ts, and a "Delete Account" link in ProfileScreen.tsx. tsc
