  refreshes. Totals refresh after edit/delete.

  Part 2 — Composite meal descriptions

  Architecture (justified): a provider-independent parser interface (parser.ts)
  with the existing whole-query USDA search as the always-on fallback, plus a
  single optional OpenAI implementation that activates only when OPENAI_API_KEY
  is set. No key → byte-for-byte the old direct-search behavior.

  New/changed edge files:
  - parser.ts — MealParser interface + OpenAiParser. Strict json_schema (no
  nutrition fields), every field re-validated (validateParsedMeal), 8s timeout,
  component cap, returns null on any failure. The prompt tells the model to keep
  real dishes ("macaroni and cheese") as one component.
  - composite.ts — deterministic gram scaling + summing of USDA nutrients.
  Unmatched components contribute nothing and raise a warning (no silent zeros);
  a fat subtype is summed only when every component reports it; portion
  assumptions and a conservative confidence range are produced.
  - index.ts — refactored into a resolveCandidates cache-or-search helper reused
  by the whole-query path and each component (bounding USDA calls, reusing
  food_search_cache/foods); parse results cached under a parse:: key namespace
  (no new migration); composite path fully wrapped so any failure falls back to
  direct candidates. Input capped at 180 chars; keys/auth headers never logged.
  - usda.ts — exported scale/round for reuse.

  Client:
  - types.ts — additively extended MealEstimateCandidate (kind, originalQuery,
  components, assumptions, warnings, confidenceRange) + ComponentEstimate.
  Missing kind ⇒ treated as direct, so older clients/cached rows keep working.
  - MealLoggerScreen.tsx — CandidateCard shows the ingredient breakdown,
  assumptions, warnings, and a confidence range before "Use & Edit". Confirming
  still routes through applyEstimate → the one logMeal() save path (source = 
  'user_estimate').
