# Profile name glitch — progress

## Root cause

The visible `user_594a…` value is the temporary username generated by the
Supabase `handle_new_user` trigger. It is not the raw user ID, though it is made
from part of that ID.

The main break was in `src/services/profileService.ts`, inside
`updateOnboardingProfile()`:

- The onboarding form collected a real name, university, and goal type, but the
  service payload only wrote `username` and macro goals. It never wrote
  `display_name`, `university`, or `goal_type`.
- If the username slug was already taken, the `23505` retry removed `username`
  and saved only the macros. That permanently left the generated `user_<id>`
  username as the only available display value.
- During signup, auth hydration and Profile-screen hydration could overlap. A
  slower, older request could therefore overwrite the store with the temporary
  username after a newer request had loaded the completed profile.

## Fix applied

- `updateOnboardingProfile()` now saves `display_name`, `university`, and
  `goal_type` along with the username and macros.
- The duplicate-username retry now omits only the unique username; it preserves
  the real display name and the rest of onboarding data.
- `getProfileIdentity()` recognizes generated `user_<id>` usernames and, for an
  existing OAuth account with no saved display name, uses the authenticated
  provider's `full_name`/`name` (or, last-resort, the email handle) instead of
  exposing the placeholder.
- Profile refreshes are sequenced so only the newest in-flight refresh can write
  to the user store.
- The signup screen now passes the collected name, university, and goal type to
  the service and refreshes once after that write completes.

## Expected result

The Profile header uses the saved human-facing name (for example,
`Nityanth Maram Reddy`). The generated username remains an internal uniqueness
fallback and is no longer shown as the person's name.
